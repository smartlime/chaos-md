# 01-cpu-load — Дефицит CPU

## Справка (`./01-cpu-load.sh -h`)

Ровно один из флагов: **`-1`** (нода), **`-4`** (основной ДЦ), **`-A`** (альтернативный ДЦ).  
Ключи: `-p` процент CPU, `-t` длительность фазы (сек), `-T` таймаут в ChaosBlade (≥ `-t`), `-D` остановка экспериментов.

```bash
./01-cpu-load.sh -1 -p 90 -t 1200 -T 1300
./01-cpu-load.sh -4 -t 600
./01-cpu-load.sh -D
```

## Как работает blade cpu load

ChaosBlade нагружает CPU, запуская параллельные горутины с бесконечным циклом.
`--cpu-percent N` ограничивает целевую загрузку: blade периодически усыпляет горутины,
чтобы удержать нагрузку в районе N%.

`--timeout N` встроен в команду blade — нагрузка снимается автоматически без внешнего таймера.

## Команды

### Шаг 2: одна нода (1200 с по умолчанию)

```bash
sudo ~/blade create cpu load --cpu-percent 90 --timeout 1200
```

### Шаг 4: весь ДЦ (600 с по умолчанию)

```bash
# Запускается параллельно на каждом из 4 хостов DC_HOSTS
sudo ~/blade create cpu load --cpu-percent 90 --timeout 600
```

Вывод blade при успехе:
```json
{"code":200,"success":true,"result":"e677f6cfa7d6644b"}
```

`result` — UID эксперимента, используется для досрочной остановки.

## Автоматическая остановка по таймауту

`--timeout N` — нагрузка снимается blade самостоятельно через N секунд.
Дополнительных nohup-обёрток не требуется.

## Досрочная остановка (-D)

UID каждого эксперимента сохраняется в `logs/01-cpu-load.<host>.uid`.

```bash
./01-cpu-load.sh -D                    # по сохранённым UID
./01-cpu-load.sh -D e677f6cfa7d6644b   # по явному UID (только для SINGLE_HOST)
```

Или вручную:
```bash
ssh <host> "sudo ~/blade destroy <uid>"
```

## Проверка

```bash
# Текущая загрузка CPU
ssh <host> "top -bn1 | head -5"

# Процессы blade
ssh <host> "pgrep -a -f 'chaos_cpu'"
```

## Параметры скрипта

| Параметр | Ключ | По умолчанию |
|---|---|---|
| Процент CPU | `-p` / `--cpu-percent` | `DEFAULT_CPU_PERCENT` (90) |
| Длительность фазы наблюдения, с | `-t` / `--time` | `DEFAULT_CHAOS_TIMEOUT` (1200) |
| Таймаут в ChaosBlade, с | `-T` / `--blade-timeout` | `DEFAULT_BLADE_TIMEOUT` (630); обычно ≥ `-t` |
| Досрочно остановить | `-D [UID]` | — |

`-t` — сколько управляющая машина **ждёт с тикером** (фаза эксперимента). `-T` — `--timeout` **внутри blade** (когда нагрузка снимается на ноде). Scope: `-1`, `-4` или `-A`.

## Что наблюдать

- Рост latency запросов YDB при высокой загрузке CPU
- Деградация на одной ноде vs. деградация при нагрузке всего ДЦ
- Срабатывание throttle/backpressure в YDB
- Время восстановления после снятия нагрузки
