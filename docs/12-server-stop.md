# Тест 12: остановка сервисов (эмуляция падения сервера)

## Справка (`./12-server-stop.sh -h`)

```
Использование: 12-server-stop.sh -1|-4|-A [ОПЦИИ]

Тест 12: остановка юнитов из env (YDBD_STORAGE_SERVICE, YDBD_TENANT_SERVICES).

Обязательно (ровно один):
  -1, --single          Одна нода
  -4, --dc              Основной ДЦ
  -A, --dc-alt          Альтернативный ДЦ

Опции:
  -t, --time SEC        Длительность «окна» остановки
  -H, --host HOST
  -C [HOST]             Статус сервисов
  -D                    Поднять сервисы досрочно
  -h, --help
```

## Назначение

Эмулирует недоступность ноды: **stop** tenant-юнитов, затем storage. Имена юнитов — в **`env.sh`** (`YDBD_STORAGE_SERVICE`, `YDBD_TENANT_SERVICES`; в примерах часто `ydbd-storage.service`, `ydbd-database-a.service`).

Порядок stop: tenant → storage. Порядок start: storage → пауза 3 с → tenant.

## Механика

1. **Apply:** на каждом хосте `systemctl stop` по списку юнитов; на ноде стартует фоновый recovery-скрипт (`nohup`), который через `-t` секунд поднимает storage, затем tenant.
2. **Ожидание:** управляющая машина ждёт `-t` с тикером (`chaos_run_window_no_teardown` — явный teardown после окна не вызывается, восстановление делает фоновый скрипт).
3. **`-D`:** убить фоновый таймер и **немедленно** выполнить start (storage → tenant).

## Параметры

| Параметр | Ключ | По умолчанию |
|----------|------|--------------|
| Длительность окна, с | `-t` | `DEFAULT_CHAOS_TIMEOUT` |
| Область | `-1` / `-4` / `-A` | один обязателен |
| Досрочный подъём | `-D` | — |
| Проверка | `-C` [HOST] | — |

## Что наблюдать

- Деградация кластера при stop одной ноды vs всего ДЦ
- Время восстановления после auto-recovery
- Корректность данных после подъёма tenant + storage
